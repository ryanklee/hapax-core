# Workspace Vision System Design

## Overview

Add dual-webcam visual awareness to the Hapax system, evolving screen-only awareness into full workspace awareness. Two cameras provide complementary views: operator-facing (BRIO 4K) for presence and attention, hardware-facing (C920) for music production gear state. A third input — the BRIO's IR sensor — enables darkness-immune presence detection.

This design covers the foundational capture primitive, five primary capability areas, and integration points across the entire Hapax system.

**Device stability:** The `by-id` symlinks (generated by the system udev rule `60-persistent-v4l.rules`) are stable across reboots and re-plugs. They incorporate the USB serial number (BRIO: `5342C819`, C920: `2657DFCF`), so even two identical models would get distinct paths. No custom udev rules required.

**Short symlinks (optional):** A `/etc/udev/rules.d/90-webcam.rules` file could create `/dev/webcam-brio`, `/dev/webcam-c920`, and `/dev/webcam-ir` for readability, using `ATTR{index}=="0"` to select only the main capture node per device.

**Camera positioning:**
- BRIO: Monitor-mounted, facing operator. Clear face and upper body visibility. Also sees MPC Live III and keyboard controller in periphery.
- C920: Elevated, angled down at desk surface. Full view of Elektron Digitakt II / Digitone II (upper left), OXI One MKII (lower left), SP-404 MKIIs (center), pedal board, keyboard controller. Good coverage of gear LEDs and display screens.

**Group membership:** The operator is not in the `video` group, but device ACLs (`+` permissions) allow access. Adding to `video` group is recommended for reliability: `sudo usermod -aG video <username>`.

## Architecture

### Design Principles

1. **Fail-open everywhere.** Camera unavailable, face detection fails, Gemini API down — all degrade gracefully. Vision is an enhancement layer, never a hard dependency.
2. **Composition over inheritance.** WebcamCapturer is a peer to ScreenCapturer, not a replacement. WorkspaceMonitor composes all capture sources.
3. **Local inference for real-time, cloud inference for reasoning.** Face detection runs locally on CPU (<5ms). Scene understanding goes to Gemini Flash (~1s). Never block on cloud calls for presence decisions.
4. **Respect VRAM.** The RTX 3090 (24GB) is shared with Ollama. All local vision inference must run on CPU or use negligible GPU. MediaPipe and frame-diff are CPU-only by design.
5. **Single-operator axiom.** No face recognition needed — there is exactly one expected operator. Face detection (is someone there?) is sufficient. Guest detection (is there more than one person?) is the only multi-face case.

### Capture Strategy

**Frame capture method:** `ffmpeg` spawned per capture. Measured at ~1.4s per frame (1080p MJPEG) on this system. For the target cadences (5-60s), this is acceptable and avoids holding exclusive device locks. If future requirements demand <5s dual-camera capture cycles, switch to OpenCV `VideoCapture` with persistent handles.

**Resolution strategy:** All captures at 1280x720 (MJPEG input). This balances quality, token cost, and processing speed:
- Gemini Flash token cost: ~516 tokens per 720p image (~2 tiles at 258 tokens each)
- Three images (screen + BRIO + C920) per analysis: ~1,548 image tokens + prompt
- Cost: ~$0.00015 per three-image analysis at Gemini 2.0 Flash pricing ($0.10/M input tokens)
- Latency: ~0.5-1.2s end-to-end for a three-image Gemini Flash call

**Capture cadences:**

| Source | Trigger | Cadence | Purpose |
|--------|---------|---------|---------|
| Screen | AT-SPI focus change or staleness | 10-60s | Digital workspace state |
| BRIO (operator) | Periodic presence check | 5-10s | Presence detection (local ML) |
| BRIO (operator) | Workspace analysis | 30-60s | Operator context for Gemini |
| C920 (hardware) | Workspace analysis | 60s or on-demand | Gear state monitoring |
| BRIO IR | Presence fallback | 10s | Darkness-immune presence |

**Key insight:** Presence detection captures (BRIO every 5-10s) are processed locally with MediaPipe (<5ms) and never leave the machine. Full workspace analysis captures (all three sources) go to Gemini Flash at a slower cadence (30-60s). This separates the fast local loop from the slower cloud loop.

## Component Designs

### 1. WebcamCapturer

**File:** `agents/hapax_voice/webcam_capturer.py`

A role-aware webcam capture primitive, parallel to the existing `ScreenCapturer`.

```python
@dataclass(frozen=True)
class CameraConfig:
    device: str          # /dev/v4l/by-id/... or /dev/video0
    role: str            # "operator", "hardware", "ir"
    width: int = 1280
    height: int = 720
    input_format: str = "mjpeg"  # mjpeg for RGB cameras, rawvideo for IR
    pixel_format: str | None = None  # "gray" for IR sensor
```

**Responsibilities:**
- Capture a single frame from a V4L2 device via `ffmpeg`
- Downscale if needed (IR sensor at 340x340 passes through as-is)
- Return base64-encoded JPEG (or PNG for IR)
- Enforce per-device cooldown (configurable, default 5s for presence, 30s for hardware)
- Verify device exists before capture attempt (fail-open: return None)

**Multi-camera management:**
- `WebcamCapturer` holds a dict of `CameraConfig` by role
- `capture(role="operator")` captures from the BRIO
- `capture(role="hardware")` captures from the C920
- `capture(role="ir")` captures from the IR sensor
- Each role has independent cooldown tracking

**Device enumeration on init:**
- Check `by-id` symlinks first (stable across reboots)
- Fall back to `/dev/videoN` enumeration if symlinks missing
- Log available cameras at startup; warn if expected cameras not found
- Never crash on missing camera

### 2. PresenceDetector Upgrade

**File:** `agents/hapax_voice/presence.py` (modify existing)

**Current state:** Silero VAD sliding window. Scores based on speech event count in last 5 minutes. Three levels: `likely_present` (5+), `uncertain` (2-4), `likely_absent` (<2).

**Enhancement:** Fuse VAD score with face detection score for a composite presence signal.

**Face detection model:** MediaPipe BlazeFace (short-range, model_selection=0).
- Runs on CPU only — zero GPU impact
- <5ms per frame at 320x320 (we'll downscale from 720p before detection)
- Works headless (no display required)
- Handles the red ambient lighting well (tested: BRIO auto-exposure produces usable frames)
- New dependency: `mediapipe` package

**IR fallback:** When the RGB face detection returns no-face for >30s, attempt IR capture. The BRIO's active IR illuminator (850nm) works in complete darkness. MediaPipe face detection works on greyscale images down to 128x128, so the 340x340 IR stream is sufficient.

**Fusion scoring:**

| VAD Score | Face Detected | Composite | Rationale |
|-----------|--------------|-----------|-----------|
| likely_present | Yes | **definitely_present** | Both signals agree |
| likely_present | No | **likely_present** | Speaking but face not visible (turned away, document held up) |
| uncertain | Yes | **likely_present** | At desk but quiet |
| uncertain | No | **uncertain** | Ambiguous — could be AFK with residual VAD |
| likely_absent | Yes | **likely_present** | Silent but physically at desk |
| likely_absent | No | **likely_absent** | Both signals agree |

**New score level:** `definitely_present` gives the ContextGate higher confidence for proactive delivery.

**Face detection loop:** Runs as a lightweight coroutine alongside the existing VAD. Captures a 320x240 frame from BRIO (or IR fallback) every 5-10s. Updates a `_face_detected: bool` flag with a 30s decay (face detected = True, stays True for 30s after last detection to handle brief look-aways).

**Privacy note:** No face recognition, no face embeddings stored, no images persisted. The detector returns a boolean (face present yes/no) and discards the frame immediately. Single-operator axiom means we never need to know *who* — only *whether*.

### 3. WorkspaceMonitor (evolved ScreenMonitor)

**File:** `agents/hapax_voice/workspace_monitor.py` (new, replaces screen_monitor.py)

The current `ScreenMonitor` composes screen capture + analysis + proactive routing. `WorkspaceMonitor` extends this to compose all visual sources: screen, operator camera, hardware camera.

**Key evolution:** The analysis call changes from single-image to multi-image. Instead of sending only a screenshot to Gemini Flash, send up to three images with role annotations.

**Analysis trigger logic:**

```
On AT-SPI focus change OR staleness timer:
  1. Capture screen (cosmic-screenshot)
  2. Capture BRIO frame (if not on cooldown)
  3. Capture C920 frame (if not on cooldown)
  4. Query RAG with previous analysis keywords
  5. Send all available frames + RAG context to WorkspaceAnalyzer
  6. Store result, route proactive issues
```

If a camera is unavailable, the analysis proceeds with whatever frames are available. The system always has at least the screenshot (existing behavior). Camera frames are additive context.

**WorkspaceAnalysis model** (extends ScreenAnalysis):

```python
@dataclass
class WorkspaceAnalysis:
    # Screen awareness (existing)
    app: str
    context: str
    summary: str
    issues: list[Issue]
    suggestions: list[str]
    keywords: list[str]
    # Operator awareness (new)
    operator_present: bool
    operator_activity: str       # "typing", "using_hardware", "reading", "away", "unknown"
    operator_attention: str      # "screen", "hardware", "away", "unknown"
    # Hardware awareness (new)
    gear_state: list[GearObservation]
    workspace_change: bool       # significant change from last observation

@dataclass
class GearObservation:
    device: str          # "MPC Live III", "Digitakt II", "SP-404 MKII", etc.
    powered: bool | None # True/False/None(can't tell)
    display_content: str # what's visible on the device screen, if anything
    notes: str           # any other observations (LED state, cable changes)
```

**Prompt engineering:** The Gemini Flash system prompt is updated to describe the three-image input format:

```
You are a workspace awareness system for a single-operator music production studio
(Linux/COSMIC/Wayland). You receive up to three images per analysis:

1. SCREENSHOT: The operator's primary monitor
2. OPERATOR CAMERA: Front-facing camera showing the operator at their desk
3. HARDWARE CAMERA: Camera facing the music production hardware rig

Return a JSON object with these fields:
[... existing screen fields ...]
- operator_present: boolean, is the operator visible in the operator camera
- operator_activity: what the operator appears to be doing
- operator_attention: where the operator's attention is directed
- gear_state: list of observed hardware devices with power/display state
- workspace_change: boolean, significant physical change from typical state

Rules:
[... existing rules ...]
- For gear_state, only report devices you can identify with reasonable confidence
- If a camera image is not provided, omit the corresponding fields
- The hardware rig includes: OXI One MKII, 2x SP-404 MKII, MPC Live III,
  Digitakt II, Digitone II, and various effects pedals
```

### 4. Hardware Awareness (C920)

The C920's primary value is music production gear monitoring. This feeds into multiple downstream systems.

**Gear state tracking:**
- On each C920 capture, Gemini Flash identifies which devices are powered on (LED indicators, lit displays, active screens)
- Results stored in `WorkspaceAnalysis.gear_state`
- Over time, builds a session log: "Digitakt turned on at 14:02, MPC at 14:05, session appears to be drum programming"

**Integration with existing systems:**

**Voice daemon context:** When a voice session opens, the system prompt includes gear state: "The operator has Digitakt II and MPC Live III powered on. The Digitakt appears to be in pattern mode." This lets the voice assistant give hardware-aware responses.

**Briefing agent:** Add a "Studio Activity" section to the daily briefing. Aggregate gear state observations over the reporting window: "You spent approximately 3 hours with the Digitakt and MPC active. The SP-404s were not used."

**Proactive awareness:** If the C920 detects a gear display showing an error state (error LED, crash screen), route through the existing proactive notification system. Example: "The MPC Live III appears to show a disk error on its display."

**Ambient audio correlation:** When the audio processor detects music activity (via PANNs classification) AND the C920 shows gear powered on, the system has high confidence the operator is making music. This composite signal could suppress all non-urgent notifications during production sessions.

### 5. Document Scanner (Hotkey-Triggered)

**Trigger:** New hotkey command via existing HotkeyServer (`agents/hapax_voice/hotkey.py`). The hotkey socket already supports arbitrary string commands.

**Flow:**
1. Operator holds document/label/note up to BRIO camera
2. Presses hotkey (e.g., `Super+Shift+V` for "visual capture")
3. HotkeyServer receives `"scan"` command
4. WebcamCapturer takes a high-res BRIO frame (1920x1080 for OCR quality)
5. Gemini Flash extracts text and structure
6. Result offered to operator via voice session or clipboard:
   - Short text (label, title): copy to clipboard via `wl-copy`
   - Structured content (patch notes, config): save to `~/documents/rag-sources/` for RAG ingestion
   - Sample names: format for SP-404 naming convention (A01-J16)

**Prompt for scan mode:**
```
Extract all text and structure from this image. The image is a photograph of
a physical document, label, or handwritten note in a music production studio.

Return JSON:
- type: "handwritten_note" | "printed_label" | "screen_photo" | "other"
- text: the extracted text content
- structured: any structured data (tables, lists, key-value pairs)
- suggested_filename: a descriptive kebab-case filename for this content
```

**No new service needed.** This is a command handler in the existing voice daemon, using the existing Gemini Flash pipeline and RAG ingestion path.

### 6. Workspace Timelapse Service

**File:** New systemd user service: `webcam-timelapse.service` + `webcam-timelapse.timer`

A lightweight continuous recording service parallel to `audio-recorder.service`. Captures one frame per minute from each camera, stores as JPEG sequence.

**Implementation:** Single ffmpeg command per camera, or a small Python script that uses WebcamCapturer on a timer.

**Storage:**
- Path: `<local-share>/hapax-voice/timelapse/`
- Naming: `{role}-{YYYYMMDD}-{HHMMSS}.jpg` (e.g., `operator-20260309-140500.jpg`, `hardware-20260309-140500.jpg`)
- Size: ~80-100KB per frame at 720p JPEG quality 85 → ~144MB/day/camera → ~288MB/day total
- Retention: 7 days rolling (configurable). Cleanup via systemd timer or the script itself.

**Value:**
- Visual log of production sessions (correlate with audio recordings by timestamp)
- Briefing agent can reference: "Based on timelapse, you were at the hardware station from 14:00-17:00"
- Teaching session documentation (when teaching on SP-404)
- Change detection baseline (frame-diff against timelapse history detects equipment moves)
- Demo agent could composite timelapses into session recap videos

**Resource impact:** Minimal. One ffmpeg invocation per minute per camera. No GPU, no ML inference. Pure I/O.

### 7. Multi-Camera Fusion: Compound Awareness

The deepest value of dual cameras comes from reasoning across all visual sources simultaneously. This section describes compound capabilities that emerge from combining signals.

**7a. Activity Mode Detection**

Fuse all signals to classify the operator's current mode:

| Screen State | BRIO (Operator) | C920 (Hardware) | Audio | Activity Mode |
|-------------|-----------------|-----------------|-------|---------------|
| IDE/terminal active | Typing on keyboard | Gear off | Silence/typing | **coding** |
| IDE/terminal active | Typing on keyboard | Gear off | Music playback | **coding_with_music** |
| Browser/docs | Reading | Gear off | Silence | **research** |
| DAW/MIDI app | Hands on hardware | Gear on, active displays | Music | **production** |
| Any | Away from desk | Any | Silence | **away** |
| Any | Present, hands idle | Gear on | Music | **listening** |
| Video call app | Facing camera | Any | Speech | **meeting** |
| Any | Multiple people | Any | Conversation | **collaboration** |

Activity mode feeds into:
- **ContextGate:** `production` mode → block all non-urgent notifications
- **Voice persona:** Adjust system prompt based on mode ("You're in a production session" vs "You're coding")
- **Briefing:** Time-in-mode statistics for daily summary
- **Notification priority:** Downgrade `normal` to `low` during `production` and `meeting` modes

**7b. Teaching Mode Detection**

When the BRIO detects multiple people (operator + guest) and the C920 shows SP-404 powered on, the system enters `teaching` mode:
- Suppress all notifications
- Voice assistant switches to simplified, family-friendly language
- If voice session opens, context includes "You appear to be teaching music production"
- Timelapse captures at higher frequency (every 30s) for session documentation

**7c. Security / Anomaly Detection**

Overnight or when the operator is known to be away:
- C920 frame-diff detects physical changes (gear moved, someone entering frame)
- BRIO IR sensor detects presence in dark room
- If `likely_absent` (VAD + face) but motion detected → alert via ntfy
- Frame-diff threshold: SSIM < 0.85 on C920 between consecutive captures = meaningful change
- Not a security system — just anomaly awareness. Notification only, no recording escalation.

**7d. Cockpit Dashboard Integration**

New cockpit API endpoint: `GET /api/workspace`

Response includes:
- Latest WorkspaceAnalysis (screen + operator + hardware state)
- Current activity mode
- Presence score (composite)
- Gear state summary
- Last capture timestamps per source

The cockpit-web dashboard gains a new "Workspace" panel showing:
- Activity mode indicator (color-coded)
- Gear state list (which devices active)
- Presence status
- Optional: latest BRIO + C920 thumbnails (privacy toggle in config)

## Dependencies

**New Python packages:**
- `mediapipe` — face detection for presence (CPU-only, ~30MB)
- `opencv-python-headless` — frame processing, SSIM, resize (may already be available via ComfyUI venv; if not, ~50MB)

**No new system packages required.** `ffmpeg`, `v4l2` kernel drivers, and ImageMagick are already installed.

**Optional:** `v4l-utils` package for `v4l2-ctl` diagnostic commands (`sudo apt install v4l-utils`).

## Migration Path

`WorkspaceMonitor` replaces `ScreenMonitor`. The migration is backward-compatible:

1. `WorkspaceMonitor` accepts all existing `ScreenMonitor` config fields
2. If `webcam_enabled=False`, behavior is identical to current `ScreenMonitor`
3. `WorkspaceAnalysis` is a superset of `ScreenAnalysis` — all existing fields preserved
4. Existing tests continue to pass (screen-only analysis still works)
5. The `ScreenMonitor` import in `__main__.py` changes to `WorkspaceMonitor`

## Cost Analysis

**Gemini Flash vision calls (primary cost):**
- Three images at 720p: ~1,548 image tokens per call
- System prompt + RAG context: ~2,000 tokens
- Output: ~500 tokens
- Total: ~4,048 tokens per analysis
- At $0.10/M input + $0.40/M output: ~$0.0006 per analysis
- At one analysis per 45 seconds, 12 hours/day: ~960 analyses/day
- **Daily cost: ~$0.58** (up from ~$0.19 for screen-only at one image per call)

**MediaPipe face detection:** Zero cost (local CPU inference).

**Timelapse storage:** ~288MB/day, ~2GB/week with 7-day retention.

**VRAM impact:** Zero. All local inference (MediaPipe, frame-diff) runs on CPU.

## Privacy Considerations

Consistent with single-operator axiom:
- No face recognition or identification — only presence detection (boolean)
- No webcam frames stored persistently by default (timelapse is opt-in)
- No frames leave the machine except to Gemini Flash for analysis (same as existing screenshot flow)
- IR sensor data processed locally, never transmitted
- Timelapse images stored locally with configurable retention
- Cockpit dashboard webcam thumbnails are opt-in (config toggle)
- No audio from webcam microphones (both cameras have mics — not used; Blue Yeti remains the sole audio source)

## Open Questions

1. **MediaPipe vs OpenCV YuNet:** Both are viable for face detection. MediaPipe is more widely used and better documented. YuNet is built into OpenCV (no extra dependency). Worth benchmarking both on actual BRIO frames with red ambient lighting before committing.

2. **BRIO IR sensor Linux behavior:** The IR LED activates when the stream is opened. There's a known Linux bug where it can get stuck on. Need to test whether opening/closing the IR device every 10s is reliable or if we should keep it open.

3. **Gemini Flash prompt stability:** The three-image prompt needs testing to verify Gemini Flash reliably distinguishes between the three labeled images and returns structured JSON for all fields. May need prompt iteration.

4. **Timelapse vs on-demand:** The timelapse service is low-cost but generates data that may not be consumed. Could start with on-demand capture only (triggered by workspace analysis) and add continuous timelapse later if the briefing agent actually uses it.

5. **C920 positioning refinement:** Current angle is good but some gear (lower-left, possibly OXI One) is partially occluded. A slightly higher mount point or wider-angle position may improve coverage. Evaluate after initial testing.
